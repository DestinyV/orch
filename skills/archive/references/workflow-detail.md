# archive 详细工作流

## 步骤1: 分析需求Spec和主规范

读取 `spec-dev/{requirement_desc_abstract}/spec/` 各文件（README.md、scenarios/、data-models.md、business-rules.md、glossary.md），提取场景数、模型、规则、术语。检查 `spec-dev/spec/` 是否存在（存在→合并，不存在→初始化）。

## 步骤2: 场景对标和差异分析

对比新场景与主规范：相同→去重 | 相关→合并为变体 | 全新→直接添加 | 特殊情况→异常场景。

## 步骤3: 规范合并

**场景库**：去重相同、合并类似、添加新的。
**数据模型**：相同字段→去重 | 新增字段→添加 | 新增模型→添加 | 冲突→标记。
**业务规则**：相同→去重 | 相关→关联 | 新→添加 | 冲突→标记。
**术语表**：相同→保持一致 | 新→添加 | 定义不同→标记。

## 步骤4: 一致性检查

输出各维度统计（已有数 + 新增数），列出需人工解决的冲突。

## 步骤5: 生成归档报告

输出到 `spec-dev/spec/archive-log.md`：
- 归档内容（需求ID、日期、场景/模型/规则/术语数）
- 合并结果（各维度新增数）
- 一致性检查结果
- 规范库当前状态（总数）

## 步骤6: 可选清理

用户多层确认后，备份并删除原需求目录。

## 合并示例

```
场景对标：user-create-order（新） → 主规范无此场景 → 添加
         user-login（已有） → 与主规范user-login相同 → 去重
数据模型：Order（新） → 主规范无 → 添加
         User（已有） → 新增email字段 → 添加到已有模型
```
