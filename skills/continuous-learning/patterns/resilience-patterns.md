# 容错与韧性模式

## 理论框架

韧性是系统在部分组件失效时仍能继续提供服务的能力。核心原则：假设一切都会失败，提前设计应对策略。

## 容错机制

| 机制 | 触发条件 | 行为 | 适用场景 |
|------|---------|------|---------|
| **重试** | 瞬态故障（超时/网络抖动） | 按策略重试 | 外部 API 调用、数据库连接 |
| **熔断** | 连续失败达到阈值 | 快速失败，不消耗资源 | 不可靠的下游服务 |
| **降级** | 服务不可用/过载 | 提供简化功能或缓存数据 | 非核心功能、推荐系统 |
| **限流** | 请求超过处理能力 | 拒绝超额请求 | 高并发入口、秒杀场景 |
| **隔离** | 资源竞争导致连锁故障 | 线程池/舱壁隔离 | 多租户、多依赖场景 |

## 重试策略标准

```json
{
  "max_retries": 3,
  "initial_delay_ms": 100,
  "max_delay_ms": 5000,
  "multiplier": 2.0,
  "retryable_errors": ["timeout", "connection_refused", "5xx"]
}
```

## 历史教训

- 无限制重试导致下游服务雪崩
- 重试未做幂等导致数据重复
- 熔断阈值设置过低导致正常请求被拒绝

## 常见遗漏

1. 未定义超时时间（默认无限等待）
2. 重试未设置最大次数（无限循环）
3. 未规划补偿策略（失败后如何恢复）
4. 未设置降级开关（无法手动干预）
