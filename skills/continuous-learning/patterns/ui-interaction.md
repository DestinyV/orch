# 交互与状态模式

## 理论框架

前端交互应遵循"即时反馈、状态明确、操作可逆"原则。每个交互操作至少覆盖 5 种 UI 状态。

## 五种 UI 状态

| 状态 | 触发条件 | 展示要求 | 常见遗漏 |
|------|---------|---------|---------|
| **Loading** | 数据加载中 | 骨架屏/Spinner | 无加载指示 |
| **Empty** | 数据为空 | 空状态图 + 引导操作 | 空白页面 |
| **Error** | 请求失败 | 错误信息 + 重试按钮 | 静默失败 |
| **Success** | 操作成功 | 数据展示 + 成功反馈 | 无反馈 |
| **Partial** | 部分成功 | 成功/失败分别标注 | 全有或全无 |

## 状态管理原则

| 原则 | 说明 | 反模式 |
|------|------|--------|
| **局部状态优先** | 状态尽可能靠近使用位置 | 全局 store 管理一切 |
| **不可变更新** | 状态更新返回新对象 | 直接修改原状态 |
| **乐观更新** | 先更新 UI 再等待确认 | 等确认后才更新 |
| **失败回滚** | 操作失败时恢复原状态 | 失败后状态不一致 |

## 表单交互

- 失焦校验（非提交时才校验）
- 防重复提交（按钮禁用 + 请求去重）
- 草稿自动保存（防止意外丢失）

## 历史教训

- 只设计了 Success 状态，其他状态空白
- 表单无防重复提交导致重复创建
- 长列表未做虚拟滚动导致卡顿

## 常见遗漏

1. 按钮无 Loading 状态导致重复点击
2. 列表无空状态提示
3. 错误信息不友好（显示技术细节）
4. 权限不足时无提示（白屏或无响应）
