# 安全防护模式

## 理论框架

安全是分层防御体系，不应依赖单一控制点。遵循纵深防御（Defense in Depth）原则。

## 认证与授权

| 层级 | 机制 | 适用场景 |
|------|------|---------|
| **身份认证** | JWT / Session / OAuth2 | 用户身份验证 |
| **访问控制** | RBAC / ABAC / ACL | 资源权限管理 |
| **操作审计** | 操作日志 + 变更追踪 | 合规要求、问题追溯 |

## 常见安全威胁与防护

| 威胁 | 防护措施 | 检查清单 |
|------|---------|---------|
| **SQL 注入** | 参数化查询 / ORM | 所有 SQL 使用占位符 |
| **XSS** | 输出转义 / CSP 头 | 所有用户输入转义后输出 |
| **CSRF** | CSRF Token / SameSite Cookie | 所有状态修改操作验证 Token |
| **越权访问** | 服务端权限校验 | 每次请求验证资源归属 |
| **暴力破解** | 限流 / 锁定 / 验证码 | 登录接口频率限制 |

## 敏感数据处理

- 密码：bcrypt/scrypt 哈希存储，明文传输使用 HTTPS
- 身份证号/手机号：展示时脱敏（如 `138****1234`）
- 日志：禁止打印敏感字段（密码、Token、身份证号）

## 历史教训

- 前端权限校验可被绕过，后端必须二次校验
- Token 未设置过期时间导致长期有效
- 敏感操作（删除/转账）无二次确认

## 常见遗漏

1. 接口未做水平越权检查（用户操作他人数据）
2. 文件上传未校验文件类型（可上传恶意脚本）
3. 错误信息泄露内部实现细节
4. 敏感接口缺少审计日志
