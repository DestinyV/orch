# 性能优化模式

## 理论框架

性能优化应遵循"测量→优化→验证"循环，避免过早优化。核心原则：先找瓶颈，再对症下药。

## 分层优化策略

| 层级 | 优化手段 | 典型收益 |
|------|---------|---------|
| **前端** | 代码分割、懒加载、虚拟列表、CDN | 首屏加载 < 3s |
| **网络** | HTTP/2、Gzip/Brotli 压缩、缓存策略 | 传输体积减少 70%+ |
| **API** | 批量查询、N+1 消除、连接池 | 响应时间减少 50%+ |
| **数据库** | 索引优化、查询重写、读写分离 | 查询时间减少 80%+ |
| **缓存** | Redis 缓存、本地缓存、CDN 缓存 | 数据库负载减少 60%+ |

## 缓存策略选择

| 场景 | 策略 | 过期时间 | 更新方式 |
|------|------|---------|---------|
| 热点读数据 | Redis 缓存 | 5min-1h | 写时失效 |
| 静态资源 | CDN 缓存 | 1 年（带 hash） | 文件名变更 |
| 配置数据 | 本地缓存 | 启动时加载 | 动态刷新 |
| 用户会话 | 分布式会话 | 30min | 自动续期 |

## 历史教训

- 未建索引导致全表扫描，响应 > 5s
- 缓存未设置过期导致数据永久不一致
- N+1 查询导致数据库连接池耗尽

## 常见遗漏

1. 未定义性能指标（P95 响应时间、吞吐量）
2. 未规划数据库慢查询监控
3. 未考虑并发用户的资源竞争
4. 未设置内存泄漏防护（连接/事件监听器未释放）
