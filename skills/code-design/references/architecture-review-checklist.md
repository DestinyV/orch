# 架构审查清单

## 理论基础

- **Clean Architecture 依赖规则**：依赖只能从外层指向内层——表现层→应用层→领域层←基础设施层。领域层为系统核心，零外部依赖
- **SOLID 原则**：审查的核心标尺——SRP（单一变化理由）、OCP（扩展开放修改封闭）、DIP（依赖抽象不依赖具体）
- **DDD 战术模式**：聚合保证事务边界、值对象不可变、仓储隔离数据访问、实体封装业务行为
- **代码坏味道（Code Smells）** (Fowler/Beck)：上帝类、长函数、紧耦合模块、过度设计

## 8 维审查

模型根据代码上下文自主选择审查维度，不机械遍历清单，聚焦高置信度问题（≥80）。

### 层依赖
核心约束：上层只依赖直接下层，禁止反向依赖和循环依赖。领域层独立于任何框架和数据库。

### 设计模式
模式应用恰当——有简单方案不用复杂模式；实体有业务行为非贫血模型；依赖注入优于服务定位器。

### 单一职责
一个类/模块只有一个变化理由。信号：函数名暗示多职责、类文件过大、改动一个功能需要修改多处。

### 依赖倒置
依赖抽象（接口）而非具体实现。外部依赖通过适配器隔离，避免直接耦合第三方库。

### 组件质量
组件可独立测试、无重复 UI 逻辑、组合非继承、模板和脚本大小合理。

### 领域模型
聚合边界清晰、值对象不可变、仓储接口在领域层、实体使用业务术语命名。

### 反模式
上帝对象（万能类）、面条代码（无结构）、紧耦合模块、金锤子（一律用一种方案）、过早抽象。

### 注释实现
严禁空函数体+注释替代实现、注释掉的代码块、只写注释不给实现。
