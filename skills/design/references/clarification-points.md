# 设计澄清点指南

design 阶段需要确认的关键维度。模型根据项目上下文自主决定各维度的澄清深度，不机械遍历。

## 理论基础

- **ADR (Architecture Decision Records)**：每个重要设计决策应记录背景、决策内容和后果，而非仅记录"选了什么"
- **架构四视图** (Kruchten 4+1)：逻辑视图（功能）、进程视图（并发/性能）、物理视图（部署拓扑）、开发视图（模块组织）+ 场景视图（用例串联）
- **非功能性需求（NFR）**：性能、安全、可扩展性、可维护性、可观测性——虽不直接产生功能，但决定了系统质量

## 7 个澄清维度

### 1. 构建与部署

探索：构建配置变更、静态资源部署策略（CDN/Nginx）、浏览器兼容性、代码分割和 Tree Shaking、Lighthouse 性能基线。

### 2. 可观测性

探索：错误追踪方案（Sentry/自研）、核心性能指标（FCP/LCP/TTFB）、用户埋点策略、告警规则定义、链路追踪需求。

### 3. 服务依赖

探索：上下游调用关系、API 端点新增/变更/废弃、接口契约变更、下游容错策略（超时/重试/降级/熔断）。

### 4. 数据库变更

探索：新增/修改/废弃表结构、字段变更和约束、索引变更、数据迁移策略、一致性要求（强一致/最终一致）。

### 5. 基础设施

探索：缓存需求（Redis 键设计/TTL）、配置变更（环境变量/配置中心）、部署变更（Docker/K8s）。

### 6. 前端 API 依赖

探索：后端 API 清单和数据结构、分页/搜索/筛选需求、错误处理和认证方式、数据模型和对象关系。

### 7. 上线与回滚

探索：上线顺序（DB迁移→配置→代码）、灰度策略（金丝雀/蓝绿/滚动）、回滚方案、监控和告警配置。

## 澄清原则

- 已有项目约定的不必重复澄清（从 project-context.md 读取）
- 与现有模式一致的不必澄清（从 code-explorer 结果判断）
- 仅差异点和新引入点需要深入澄清
