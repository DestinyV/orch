# 需求深度追问框架

场景拆解前的需求澄清与闭环验证。

## 理论基础

需求澄清基于以下原则：

- **JTBD (Jobs-to-Be-Done)**：关注用户目标而非功能清单——理解用户雇佣产品完成的"任务"
- **INVEST 原则**：需求应 Independent（独立）、Negotiable（可协商）、Valuable（有价值）、Estimable（可估算）、Small（小粒度）、Testable（可测试）
- **5W1H 分析法**：Who（谁）、What（什么）、Why（为什么）、Where（哪里）、When（何时）、How（如何）
- **闭环思维**：需求生命周期从提出到验证满意形成闭环，不终止于上线

## 10 维度探索

每个维度提供**探索方向**和**理论基础**，不约束具体问法。模型应根据需求上下文自主决定追问路径。

### 1. 用户与干系人

**理论**：干系人分析（Stakeholder Analysis），RACI 矩阵

| 探索方向 | 常见盲区 |
|---------|---------|
| 识别所有角色及其目标差异 | 仅关注提出需求的人，忽略沉默干系人 |
| 区分决策者、使用者、受影响者 | 审批人与实际使用者脱节 |
| 权限分级与角色边界 | 所有角色平等对待，遗漏权限模型 |

### 2. 核心意图

**理论**：JTBD 核心问题——"用户雇佣这个产品完成什么任务？"；5 Whys 根因分析

| 探索方向 | 常见盲区 |
|---------|---------|
| 用户目标（不是功能列表，是待完成的任务） | 把用户说的方案当成需求本身 |
| 当前替代方案及痛点 | 不了解现状，无法判断改进价值 |
| 问题本质是新问题还是旧问题的新解法 | 将改进误判为全新需求 |

### 3. 成功标准

**理论**：OKR（目标与关键结果）、SMART 目标（Specific, Measurable, Achievable, Relevant, Time-bound）

| 探索方向 | 常见盲区 |
|---------|---------|
| 可衡量的完成标准（非主观感受） | "能用就行"无验收依据 |
| 区分发布标准与效果标准 | 上线即完成，不追踪使用效果 |
| 最小可接受标准（MoSCoW：Must/Should/Could/Won't） | 无优先级导致范围蔓延 |

### 4. 范围边界

**理论**：MoSCoW 优先级法、Kano 模型（基本型/期望型/兴奋型需求）

| 探索方向 | 常见盲区 |
|---------|---------|
| 明确 OUT of scope（比 IN scope 更重要） | 不定义边界，无限蔓延 |
| 版本拆分策略（MVP → 迭代） | 第一版承载过多 |
| 与现有功能的关系（替代/增强/新增/整合） | 忽略对已有功能的影响 |

### 5. 数据与规模

**理论**：领域驱动设计（DDD）——数据实体识别；容量规划

| 探索方向 | 常见盲区 |
|---------|---------|
| 数据实体生命周期（创建→使用→归档→销毁） | 只考虑存储，不考虑治理 |
| 规模估算（日增量/总量/峰值QPS/并发数） | 全量加载、无分页、无限流 |
| 合规要求（数据保留期/脱敏/审计追溯） | 敏感数据无保护措施 |

### 6. 技术约束

**理论**：架构决策记录（ADR）；康威定律——系统设计反映组织沟通结构

| 探索方向 | 常见盲区 |
|---------|---------|
| 技术栈边界（语言/框架/平台/协议） | 引入项目不支持的技术 |
| 兼容性矩阵（浏览器/设备/OS/API版本） | 特定环境崩溃 |
| 外部依赖图谱（服务/API/中间件/第三方） | 依赖遗漏导致后期阻塞 |

### 7. 非功能性需求

**理论**：ISO 25010 软件质量模型——性能效率/安全性/兼容性/可用性/可靠性/可维护性/可移植性

| 探索方向 | 常见盲区 |
|---------|---------|
| 性能（响应时间/吞吐量/资源消耗） | "越快越好"无量化 |
| 安全（认证/授权/加密/审计/防注入） | 敏感操作无权限校验 |
| 可访问性（WCAG 等级/屏幕阅读器/键盘导航） | 无障碍合规遗漏 |
| 国际化（多语言/时区/货币/格式） | 硬编码文案和格式 |

### 8. 时间与优先级

**理论**：艾森豪威尔矩阵（紧急/重要）；关键链项目管理（CCPM）

| 探索方向 | 常见盲区 |
|---------|---------|
| 交付时间约束（硬 deadline vs 期望日期） | 临时发现是硬 deadline |
| 依赖链里程碑（外部接口/上下游/审批节点） | 外部阻塞导致整体延期 |
| 与其他需求的优先级关系 | 资源冲突无预案 |

### 9. 异常与边界

**理论**：防御性设计（Defensive Design）；墨菲定律——可能出错的一定会出错

| 探索方向 | 常见盲区 |
|---------|---------|
| 故障模式识别（最可能出错的 3 个环节） | 仅设计正常路径 |
| 降级策略（系统不可用时的用户期望） | 无降级，白屏或崩溃 |
| 边界条件（空值/极大/极小/并发/超时） | 边界未定义行为 |

### 10. 闭环验证

**理论**：PDCA 循环（Plan-Do-Check-Act）；Build-Measure-Learn 精益循环

| 探索方向 | 常见盲区 |
|---------|---------|
| 验证责任人、周期和判定标准 | 无人负责验证 |
| 不达预期的处理策略（回滚/修复/迭代） | 无退出方案 |
| 发布策略（灰度/金丝雀/A/B测试/全量） | 全量发布后无法回滚 |
| 配套准备（文档/培训/通知/数据迁移） | 功能上线无人会用 |

## 使用指导

- **探索方式**：模型根据需求上下文自主选择各维度的追问深度和路径，不机械遍历
- **核心约束**：维度 1-4 必须覆盖（定义问题的 WHOWHATWHY 和边界），5-10 按需深入
- **记录格式**：输出到 requirement.md「确认记录」章节，每维度标注 `[已澄清/待确认/N/A]`
- **闭环判定**：全部必选维度标记「已澄清」或「N/A」后，方可进入场景拆解

## 快速模式

仅覆盖维度 1-4（用户/意图/成功标准/范围），其余标注「快速跳过」。
