# 场景拆解指南

从需求到 BDD 场景的系统化拆解方法论。

## 理论基础

- **用户故事映射（User Story Mapping）**：Jeff Patton 提出，按用户旅程的先后顺序排列场景，形成端到端的叙事地图
- **状态机建模（State Machine）**：将数据实体的生命周期建模为有限状态机，每个状态转换对应一个场景 Case
- **防御性设计（Defensive Design）**：假设每个交互都可能失败，以异常路径为第一公民而非事后补充
- **依赖图（Dependency Graph）**：场景间存在数据/状态依赖，必须建模为 DAG 以支持并行执行

## 8 维度拆解

每个维度提供**探索方向**和**理论基础**，模型根据需求上下文自主决定拆解粒度。

### 1. 角色/用户类型

**理论**：角色建模（Actor Modeling）——UML 用例图中的 Actor 概念；每个角色有独立的目标集

探索方向：识别系统边界外的所有交互角色（人类/外部系统/定时任务），区分其目标和权限差异。

### 2. 业务流程

**理论**：用户故事映射（User Story Mapping）——按时间线排列用户行为，区分主流程和替代流程

探索方向：从触发点到目标达成的完整路径，识别分支点、决策点和可选路径。覆盖 4 类流程：主流程（Happy Path）、替代流程（Alternative）、中断流程（Interrupt）、重试流程（Retry）。

### 3. 数据状态

**理论**：状态机（Finite State Machine）——实体从创建到销毁经历有限状态，状态转换由事件触发

探索方向：为每个核心数据实体建模生命周期（创建→活跃→变更→归档→销毁），识别合法与非法的状态转换，覆盖空/无数据的边界状态。

### 4. 异常与边界

**理论**：防御性设计（Defensive Design）——墨菲定律；契约式编程（Design by Contract）——前置条件/后置条件/不变式

探索方向：假设每个交互都可能失败，从输入边界（空/极值/非法格式）、权限边界（越权/未认证）、资源边界（不存在/已删除/并发冲突）、外部依赖（超时/不可用）四个方向识别异常。

### 5. 权限边界

**理论**：RBAC（基于角色的访问控制）——许可与角色关联；最小权限原则

探索方向：区分每个角色的可见性和可操作性，验证横向越权（用户操作他人数据）和纵向越权（低权限执行高权限操作）。

### 6. 时空条件

**理论**：分布式系统的时间不确定性——Lamport 逻辑时钟；事件最终一致性

探索方向：跨时区场景、过期/截止时间行为、定时/延迟操作、离线状态与网络恢复后的数据同步。

### 7. 前后端交互

**理论**：API 状态机——每个 API 调用存在 5 个 UI 状态；前后端契约

探索方向：前端每个交互覆盖 Loading → Empty → Error → Success → Partial 五种 UI 状态的展示和切换。

### 8. 数据一致性

**理论**：CAP 定理——一致性/可用性/分区容错不可兼得；BASE 理论——基本可用/软状态/最终一致性

探索方向：跨服务操作的原子性边界、异步处理的消息可靠性、缓存与数据库的一致性偏差场景。

## 场景依赖

场景间存在数据/状态依赖，应建模为有向无环图（DAG）以支持并行执行和调度。

每个场景文件头部声明：
- `depends-on`：本场景执行前必须完成的场景
- `provides-to`：依赖本场景输出的下游场景

依赖图应保证无环——循环依赖意味着需求分析存在缺陷。
