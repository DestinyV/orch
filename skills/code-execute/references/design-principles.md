# 软件设计原则

> code-execute 阶段 REFACTOR 和代码审查参考

## SOLID

| 原则 | 含义 | 示例 |
|------|------|------|
| **S** - 单一职责 | 一个类/函数只有一个变化理由 | ❌ 函数既验证又保存 ✅ 分离 |
| **O** - 开闭 | 对扩展开放，对修改关闭 | ❌ if-else 分支 ✅ 策略模式 |
| **L** - 里氏替换 | 子类可替换父类 | ❌ 子类抛异常 ✅ 遵守契约 |
| **I** - 接口隔离 | 小接口优于大接口 | ❌ 胖接口 ✅ 拆分为小接口 |
| **D** - 依赖倒置 | 依赖抽象而非具体 | ❌ 依赖实现类 ✅ 依赖接口 |

## 其他关键原则

| 原则 | 含义 |
|------|------|
| **DRY** | 不要重复自己 |
| **KISS** | 保持简单愚蠢 |
| **YAGNI** | 你不会需要它（不做过度设计） |
| **组合>继承** | 优先组合，谨慎继承 |

## 代码异味

长函数/长类 | 过多参数 | 重复代码 | 魔法数字 | 深层嵌套 | 大上帝对象

## REFACTOR 检查

每次重构：测试仍通过 | 职责更清晰 | 命名更准确 | 无重复 | 依赖更清晰
