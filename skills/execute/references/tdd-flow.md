# TDD 流程 — 带出口验证的执行循环

## 四阶段循环（每阶段有出口验证标准）

| 阶段 | 操作 | 出口验证 | 阻塞动作 |
|------|------|---------|---------|
| **RED** | 选一个 TC 写断言，运行确认失败 | 测试失败，失败信息明确表达期望值 vs 实际值 | 若测试通过(未真正测试) → 重写断言 |
| **GREEN** | 写最少代码让测试通过 | 目标测试通过，其他测试未回归 | 若写过多代码 → 回退只保留最少实现 |
| **REFACTOR** | 提取重复/改进命名/优化结构 | 全部测试仍通过，代码可读性提升 | 若测试破裂 → 回退重构 |
| **REVIEW** | Lint + 类型检查 + 覆盖率 ≥ 85% | 0 lint, 0 type 错误, 覆盖达标, 无伪代码 | 任一项不满足则阻塞修复后重检 |

## 门控规则

<GATE>RED 未通过(测试未写或测试通过而非失败) → 禁止进入 GREEN</GATE>
<GATE>GREEN 未通过(测试仍失败) → 禁止进入 REFACTOR</GATE>
<GATE>REVIEW 未通过(任一检查失败) → 该 Task 视为未完成</GATE>

## 关键原则

- 一次一个测试 | RED 必须失败 | GREEN 只做测试需要的
- REFACTOR 每次只改一个点 | Mock 只 Mock 外部依赖
- 任一阶段不满足出口验证 → 回退当前阶段，不进入下一阶段
