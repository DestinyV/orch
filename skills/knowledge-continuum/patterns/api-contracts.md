# 接口契约模式

## 理论框架

接口契约是系统间通信的协议定义。良好的接口契约应稳定、版本可控、向后兼容。

## 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "trace_id": "abc123"
}
```

## 错误码规范

| 错误码范围 | 含义 | 示例 |
|-----------|------|------|
| 0 | 成功 | |
| 1xxx | 参数错误 | 1001=必填字段缺失 |
| 2xxx | 认证/权限错误 | 2001=未认证, 2003=权限不足 |
| 3xxx | 业务规则错误 | 3001=状态不允许操作 |
| 5xxx | 系统错误 | 5000=内部服务异常 |

## 分页查询标准

**请求参数**：`?page=1&size=20&sort=created_at&order=desc`
**响应格式**：
```json
{
  "data": { "items": [], "total": 100, "page": 1, "size": 20 }
}
```

## 接口设计原则

| 原则 | 说明 |
|------|------|
| **RESTful 风格** | 资源命名用名词复数，操作用 HTTP 方法 |
| **幂等性** | GET/PUT/DELETE 必须幂等 |
| **版本控制** | URL 路径或 Header 中包含版本号 |
| **限流保护** | 批量操作限制最大数量（如 100 条/次） |

## 历史教训

- 列表接口未返回 total 导致前端无法分页
- 未定义幂等性导致重复提交产生脏数据
- 接口字段类型前后端不一致（字符串 vs 数字）

## 常见遗漏

1. 未定义接口超时时间和重试策略
2. 未规定请求频率限制（防刷）
3. 敏感接口未做防重放攻击
4. 文件上传未限制大小和类型
5. 未定义分页参数的默认值和最大值
